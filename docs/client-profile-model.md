# Модель «клиент у психолога»: один человек, несколько психологов, опциональный аккаунт

## Идея

- **Добавление клиента ≠ клиент обязан заходить в CRM.** Психолог может вести клиентов, которые никогда не регистрируются.
- **Связка по email.** Если клиент с указанным email потом зарегистрируется — его аккаунт (User) связывается с профилем клиента у психолога (ClientProfile).
- **Один клиент — много психологов.** У одного человека (один email / один User) может быть несколько ClientProfile — по одному на каждого психолога.
- **Изоляция профилей.** Редактирование профиля клиента у психолога A не должно менять:
  - профиль того же клиента у психолога B;
  - «личный» профиль клиента (если он сам завёл аккаунт).
- **Общее у клиента и психолога:** записи (Appointment) и психологические диагностики (TestResult и т.п.) — они привязаны к паре «психолог + клиент» (через ClientProfile).

---

## Нюансы и решения

### 1. Добавление клиента без аккаунта

- Психолог может добавить клиента **без создания User**: только имя, фамилия, телефон, заметки и **email «для связи»** (опционально).
- В БД создаётся только **ClientProfile** с `psychologistId`, без `userId` (или с `userId = null`).
- Email для будущей связки нужно хранить **в ClientProfile** (поле `email`, nullable), чтобы при регистрации клиента по этому email можно было привязать его к этому профилю.

**Итог:** в схеме у ClientProfile должно быть поле `email String?` (нормализованное: trim, toLowerCase). При создании клиента «без аккаунта» User не создаём; при создании «с аккаунтом» (или при связке по email) — находим/создаём User и проставляем `userId` в ClientProfile.

---

### 2. Один человек — один User, много ClientProfile

- **User** — один на человека (ключ — email).
- **ClientProfile** — один на пару (психолог, клиент). У одного User может быть несколько ClientProfile: у разных психологов и, при желании, один «личный» (psychologistId = null) для данных, которые клиент заполнил сам.

**Схема:**

- В `ClientProfile`: `userId String?` (опционально), убрать `@unique` с `userId`, чтобы несколько профилей могли ссылаться на одного User.
- В `User`: связь `clientProfiles ClientProfile[]` (вместо одной `clientProfile`).
- В `ClientProfile`: поле `email String?` для хранения email, когда `userId` ещё нет (для последующей связки).

---

### 3. Нормализация email

- При связке и поиске дубликатов использовать **нормализованный** email: `trim().toLowerCase()`.
- Тогда «Client@Mail.Ru» и «client@mail.ru» считаются одним и тем же человеком; при регистрации все ClientProfile с таким email будут привязаны к одному User.

---

### 4. Изоляция данных профиля

- У каждого ClientProfile свои поля: `firstName`, `lastName`, `phone`, `notes`, `dateOfBirth` и т.д.
- Обновления делаем **только по id ClientProfile** (и проверяем, что psychologistId совпадает с текущим психологом). Никогда не обновляем «все профили по userId» — иначе затронем другого психолога или «личный» профиль клиента.
- Email при наличии аккаунта: для отображения у психолога лучше брать из User (источник правды для входа). Менять email входа клиента при редактировании карточки у психолога не нужно — только показывать (read-only). Когда у клиента ещё нет аккаунта — редактируем и храним email в ClientProfile.

---

### 5. Регистрация клиента после того, как его добавили

- В момент регистрации (или привязки email к User): по `user.email` (нормализованному) найти все ClientProfile, где `email` совпадает и `userId` пустой; проставить им `userId = user.id`.
- Так все «ожидающие» профили у разных психологов автоматически свяжутся с одним аккаунтом.

**Место вызова:** после успешного создания User в `POST /api/auth/register` (для роли client) и при любом сценарии, где у существующего User впервые появляется/меняется email (если такое будет).

---

### 6. Смена email у клиента (User)

- Если клиент меняет email в настройках аккаунта — меняется только `User.email`. ClientProfile.email можно не трогать (оставить старый для истории) или не хранить после связки.
- Политика: после установки `userId` у ClientProfile считаем «источником правды» для email именно User. В карточке у психолога показываем User.email (read-only). Старое значение в ClientProfile не используем для отображения, только для истории/аудита при желании.

---

### 7. Диагностики и записи — общая «точка соприкосновения»

- **Записи (Appointment):** уже привязаны к `clientId` (ClientProfile) и `psychologistId`. Один ClientProfile — один психолог, изоляция по умолчанию.
- **Диагностики (TestResult, DiagnosticLink и т.д.):** привязаны к `clientId` (ClientProfile) и/или `psychologistId`. То есть они «общие» в смысле «клиент + психолог», а не «все психологи клиента». Данные одного психолога не смешиваются с данными другого.

Для клиента с одним аккаунтом и несколькими психологами: в личном кабинете можно показывать записи и диагностики **по каждому психологу отдельно** (фильтр по psychologistId или по списку «мои психологи»).

---

### 8. «Личный» профиль клиента (self-service)

- Если клиент сам регистрируется, можно создавать один ClientProfile с `psychologistId = null` и `userId = user.id` — для имени, контактов и т.д., которые он редактирует сам.
- Альтернатива: хранить такие данные только в User (name и т.д.) и не заводить отдельный ClientProfile без психолога. Тогда «профиль клиента» существует только в связке с психологом, а в личном кабинете показываются только записи/диагностики по психологам.

Нюанс: если завести и «личный» ClientProfile, и профили у психологов — при отображении «имени» в интерфейсе психолога всегда брать имя из **своего** ClientProfile, а не из User и не из «личного» профиля.

---

### 9. Один психолог — один «слот» на одного человека (по email)

- Имеет смысл правило: у одного психолога не может быть двух разных ClientProfile с одним и тем же email (чтобы не было дублей «Иванов» и «Иванов (другой)» с одним email). При добавлении клиента с таким email — либо обновляем существующий профиль (если это явно «тот же» клиент), либо возвращаем ошибку/подсказку.
- Уникальность: `@@unique([psychologistId, email])` при ненулевом email (и при ненулевом psychologistId). Для «личного» профиля (psychologistId = null) уникальность по email можно не навязывать или обрабатывать отдельно.

---

### 10. Удаление/отвязка клиента у психолога

- При «удалении» клиента у психолога: не удалять User, не трогать ClientProfile других психологов. Либо ставить у профиля `psychologistId = null` (как сейчас), либо помечать как «архивный». Записи и диагностики по этому психологу остаются привязаны к тому же ClientProfile (история сохраняется).

---

## Краткая сводка изменений в коде/схеме

| Что | Сейчас | Нужно |
|-----|--------|--------|
| Связь User ↔ ClientProfile | 1:1 (`userId` @unique) | 1:N (у User много ClientProfile) |
| ClientProfile.userId | обязательный | опциональный (null = «без аккаунта») |
| ClientProfile.email | нет | добавить `email String?` (нормализованный) |
| Добавление клиента | всегда создаётся User | разрешить создание только ClientProfile (без User), с email для связки |
| Регистрация клиента | создаётся User + один ClientProfile | после создания User — привязать все ClientProfile с тем же email |
| Редактирование профиля | только по id ClientProfile | без изменений; не обновлять по userId |
| Уникальность | — | при необходимости: один email на одного психолога в ClientProfile |

После внедрения этих правил модель будет соответствовать описанному подходу: точка связки — email; общее для клиента и психолога — записи и диагностики; профили у разных психологов и «личный» профиль изолированы.
