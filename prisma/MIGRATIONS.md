# Миграции Prisma

## Текущая стратегия: одна миграция «с нуля»

История миграций сведена к **одной** миграции `20260315120000_init_from_schema`, которая создаёт всю схему БД из текущего `schema.prisma`. Так мы избавились от накопления «заплаточных» миграций (dateOfBirth, UNSPECIFIED, колонки профилей и т.д.) и расхождений между схемой и продом.

## Новое окружение или пересоздание БД с нуля

**Локальная разработка:**

```bash
# Полное пересоздание БД (удалит все данные)
npx prisma migrate reset

# Или пустая БД: просто применить миграции
npx prisma migrate deploy
```

**Продакшен (Vercel/Neon и т.п.):**

1. Если данные не нужны: в панели БД удалите все таблицы (или пересоздайте базу), затем при деплое выполнится `prisma migrate deploy` — применится одна миграция.
2. Если данные нужны: такую «одну миграцию с нуля» на уже существующую БД применять нельзя (она создаёт таблицы заново). Варианты:
   - Либо оставить текущую прод-базу как есть и **не** переходить на одну миграцию в проде (оставить старые миграции в репозитории только для этого окружения),  
   - Либо один раз пометить миграцию как уже применённую:  
     `npx prisma migrate resolve --applied 20260315120000_init_from_schema`  
     (выполнять с продовым `DATABASE_URL`, **не** запуская саму миграцию), затем деплоить код с одной миграцией.

## Дальнейшие изменения схемы

После перехода на одну миграцию все новые изменения делайте через новые миграции:

```bash
# После правок в schema.prisma
npx prisma migrate dev --name описание_изменения
```

Так будет одна базовая миграция + по одной миграции на каждое изменение, без бесконечных «заплаток».
